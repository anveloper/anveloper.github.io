# 커밋 메시지 가이드라인

이 프로젝트는 명시적인 커밋 히스토리를 생성하기 위해 Conventional Commits 명세를 따릅니다. 이는 프로젝트 히스토리를 살펴볼 때 따라가기 쉬운, 더 읽기 좋은 메시지를 만듭니다.

## 형식

각 커밋 메시지는 **헤더(header)**, **본문(body)**, **꼬리말(footer)**로 구성됩니다.

```
<타입>: <제목> (<스코프>)
<빈 줄>
<본문>
<빈 줄>
<꼬리말>
```

**헤더**는 필수이며 **타입**, 선택적인 **스코프**, **제목**을 포함하는 특별한 형식을 가집니다.

### 타입 (Type)

다음 중 하나여야 합니다:

*   **feat**: 새로운 기능
*   **fix**: 버그 수정
*   **docs**: 문서만 변경
*   **style**: 코드 의미에 영향을 주지 않는 변경 (공백, 서식, 세미콜론 누락 등)
*   **refactor**: 버그를 수정하거나 기능을 추가하지 않는 코드 변경
*   **perf**: 성능을 향상시키는 코드 변경
*   **test**: 누락된 테스트를 추가하거나 기존 테스트를 수정
*   **chore**: 빌드 프로세스나 문서 생성과 같은 보조 도구 및 라이브러리 변경
*   **revert**: 이전 커밋으로 되돌리기

### 스코프 (Scope)

스코프는 추가적인 문맥 정보를 제공하며 괄호 안에 포함됩니다. 예: `feat(parser): 배열 파싱 기능 추가`. 이는 형식의 선택적인 부분입니다.

### 제목 (Subject)

제목은 변경 사항에 대한 간결한 설명을 포함합니다:

*   명령형, 현재 시제 사용: "changed"나 "changes"가 아닌 "change"
*   첫 글자를 대문자로 쓰지 않음
*   끝에 마침표(.)를 찍지 않음

### 본문 (Body)

본문에는 변경의 동기와 이전 동작과의 차이점을 포함해야 합니다. 선택 사항입니다.

### 꼬리말 (Footer)

꼬리말에는 **주요 변경 사항(Breaking Changes)**에 대한 모든 정보가 포함되어야 하며, 이 커밋이 **해결(Closes)**하는 GitHub 이슈를 참조하는 곳이기도 합니다.

**주요 변경 사항**은 `BREAKING CHANGE:`라는 단어와 공백 또는 두 개의 줄 바꿈으로 시작해야 합니다. 커밋 메시지의 나머지 부분은 변경 사항에 대한 설명, 정당성 및 마이그레이션 노트입니다.

## 예시

### 제목만 있는 커밋 메시지
```
feat: 다크 모드 토글 버튼 추가
```

### 스코프가 있는 커밋 메시지
```
feat: 새 버튼 컴포넌트 추가 (ui)
```

### 본문과 주요 변경 사항이 있는 커밋 메시지
```
feat: 제공된 설정 객체가 다른 설정을 확장하도록 허용

BREAKING CHANGE: 설정 파일의 `extends` 키는 이제 다른 설정 파일을 확장하는 데 사용됩니다.
```

## 브랜치 전략

- 주요 작업 브랜치는 `develop` 브랜치를 사용합니다. 모든 기능 개발 및 수정은 `develop` 브랜치를 기준으로 합니다.
- 현재 브랜치가 `develop`이 아닌 경우, `develop`으로 변경하고, 새로운 브랜치를 생성합니다.

## 커밋 그룹화

- 커밋 그룹화 요청 시, 현재 브랜치의 전체 커밋 내역을 확인하고 분석합니다.
- 분석된 내용을 바탕으로, 관련된 커밋들을 하나의 큰 커밋으로 합치기보다는, **'기능 구현', '리팩토링' 등 여러 개의 논리적인 그룹으로 나누어 정리하는 것**을 우선합니다.

## 개발 일지 (Development Log)

- 하루 동안의 작업이 끝나면, 당일 커밋한 내역을 `docs/logs/YYYY-MM-DD.md` 형식의 파일로 생성하여 저장합니다.
- 파일 안의 내용은 시간 순서로 나열하는 것이 아니라, **신규 기능(feat), 버그 수정(fix) 등**의 커밋 타입에 따라 그룹화하여 정리합니다.
- 이미 해당 날짜의 로그 파일이 존재하는 경우, 새로운 파일을 만들지 않습니다. 대신 기존 파일의 내용에 새로운 커밋 내역을 항목, 타입, 기능에 맞게 추가합니다.
